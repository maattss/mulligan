# Offline-First Golf Scoring App With Local Course Catalog

## Summary
- Revert to the simpler original architecture: **Vue 3 on Vite**, **Pinia**, **Dexie**, and **PWA** behavior, with **no runtime backend** and **no live API dependency** in the shipped app.
- Keep **`shadcn/vue`** for UI, but use it on the **Vite** path instead of introducing Nuxt or a server layer.
- Populate the built-in course catalog from **[GolfCourseAPI](https://golfcourseapi.com/)** through a **developer-only import script**, not through runtime integration.
- The app ships with a local seeded catalog for the Bergen/Stavanger courses you care about, and competitions snapshot course data locally so scoring works fully offline.

## Key Changes
- Frontend stack:
  - Vue 3 + TypeScript
  - Vite
  - Vue Router
  - Pinia
  - Dexie
  - `vite-plugin-pwa`
  - `shadcn/vue` + Tailwind CSS v4
- Product scope stays focused on scoring and competitions:
  - Individual stroke play
  - Individual Stableford
  - Individual match play
  - Four-ball stroke play
  - Four-ball Stableford
  - 2-player scramble
  - Skins as an optional side game on stroke-based rounds, with `gross` and `net` modes
- Keep the original operating model:
  - one scorer device
  - no accounts in v1
  - per-player tees on the same course
  - manually entered Handicap Index per player
- Best ball remains covered through the four-ball formats and is not modeled as a separate side-game system.

## Course Catalog Workflow
- Do not call GolfCourseAPI from the browser at runtime.
- Add a developer-only sync script, for example `scripts/sync-courses.mjs`, that:
  - reads a curated manifest of target clubs/courses
  - calls GolfCourseAPI using a local env var such as `GOLF_COURSE_API_KEY`
  - fetches the required courses and tee data
  - normalizes the results into the app's internal course schema
  - writes a static JSON artifact into the repo, for example `src/data/course-catalog.json`
- Maintain a small source manifest, for example `src/data/course-manifest.json`, listing:
  - target Bergen/Stavanger clubs
  - preferred course identifiers
  - any course aliases needed for matching
  - whether a course should be included by default
- Normalize imported provider data into these internal types:
  - `CourseSearchSeed`
  - `CourseDetail`
  - `TeeSet`
  - `HoleDetail`
  - `CourseSnapshot`
- Required `TeeSet` fields:
  - `name`
  - `color`
  - `par`
  - `courseRating`
  - `slopeRating`
  - `yardage`
  - `holePars[]`
  - `strokeIndexes[]`
- If GolfCourseAPI returns incomplete Norwegian data for a course, keep a manual override file in-repo, for example `src/data/course-overrides.json`, to patch missing tee names, pars, ratings, or stroke indexes before the final catalog is emitted.
- At competition creation time, copy the selected course and tee data into a `CourseSnapshot` so finished rounds never change if the seeded catalog is refreshed later.
- The shipped app only reads local course data from the bundled catalog and Dexie cache.

## Rules And Interfaces
- `CompetitionFormat = 'stroke' | 'stableford' | 'match-play' | 'fourball-stroke' | 'fourball-stableford' | 'scramble-2'`
- `SideGameType = 'skins'`
- `SkinsMode = 'gross' | 'net'`
- `Competition` stores:
  - `holes`
  - `format`
  - `courseSnapshot`
  - `allowanceRuleSnapshot`
  - `players`
  - `sides`
  - `sideGames`
  - `status`
- `CompetitionPlayer` stores:
  - player name snapshot
  - handicap index snapshot
  - tee snapshot
  - course handicap snapshot
  - playing handicap snapshot
  - side assignment
- Handicap rules:
  - calculate Course Handicap from tee `courseRating`, `slopeRating`, and `par`
  - apply saved format allowance to derive Playing Handicap
  - match play uses relative strokes from the lowest Playing Handicap
  - net skins uses saved stroke allocation for the competition
- Default allowance policy:
  - Individual stroke play: 100% default, editable
  - Individual Stableford: 100% default, editable
  - Individual match play: 100%
  - Four-ball stroke play: 85%
  - Four-ball Stableford: 85%
  - 2-player scramble: 35% low / 15% high

## Test Plan
- Unit test handicap math across:
  - mixed tees
  - plus handicaps
  - 9-hole and 18-hole rounds
  - rounding edge cases
- Unit test every competition format and skins with fixed scorecards and expected outputs.
- Test the course import pipeline with recorded GolfCourseAPI fixtures so provider-shape changes are caught without needing live API calls in CI.
- Test catalog generation rules:
  - manifest entry resolves to the correct course
  - overrides patch missing data
  - normalized output always contains the required tee and hole fields
- Component test the setup flow:
  - choose built-in course
  - assign tees per player
  - start competition
  - reload and resume from local storage
- End-to-end test the mobile flow:
  - create competition from the bundled course catalog
  - enter scores
  - finish round
  - reload offline
  - review results and skins totals

## Assumptions And Defaults
- GolfCourseAPI is used only as a maintainer data source for seeding the local catalog, not as part of the runtime app architecture.
- A local API key for the sync script is acceptable, but no key or provider dependency ships in the production frontend.
- The initial seeded catalog focuses on Bergen/Stavanger and can expand later by editing the course manifest and re-running the sync script.
- Manual override files are acceptable for any Norwegian courses where GolfCourseAPI coverage is incomplete.
- The implementation document should live at `plans/implementation-plan.md` once execution starts.
